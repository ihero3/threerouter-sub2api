# AI Governance & Compliance Module - AI 治理与合规模块

**商业定位**：Sub2API AI Governance Layer 为企业使用第三方 AI 模型提供调用审计、风险识别、合规评估和跨境 AI 服务证据管理能力。

**核心价值**：Sub2API 不一定是被监管的 AI 系统，但它可以成为被监管 AI 系统的合规基础设施。

### 0.2 必须遵循的技术约定（规避陷阱）

1. **数据库迁移采用编号 SQL 文件双轨制，非 ent 自动迁移**
   - 本方案中所有以 ent schema 代码给出的字段/表，实际实施须：① 新增编号 SQL 迁移文件（如 `158_compliance_fields.sql`）；② 同步修改 service 层 struct（[service/usage_log.go](file:///d:/Users/lfh/source/repos/threerouter/threerouter-sub2api/backend/internal/service/usage_log.go#L99-L185)）；③ 修改 repo 层手写的 insert 列名（[usage_log_repo.go](file:///d:/Users/lfh/source/repos/threerouter/threerouter-sub2api/backend/internal/repository/usage_log_repo.go)）。
   - `usage_logs` 是**分区表**，新增字段和索引需注意分区兼容性，并发索引用 `_notx.sql` 后缀文件。

2. **ContentModerationService 已存在，须扩展而非新建**（见 0.3 修订）。

3. **API 前缀避开 `/admin/compliance/*`**：现有 `AdminComplianceGuard` 中间件仅放行 `/api/v1/admin/compliance*` 路径（管理员部署合规声明未确认时的豁免），若新模块沿用该前缀会导致新接口被意外放行、且与现有 `compliance_handler.go`（仅做部署合规确认）命名冲突。**新模块统一使用 `/api/v1/admin/governance/*` 前缀，Handler 命名 `GovernanceHandler`**，与"AI Governance Layer"定位一致。

4. **GeoIP 为全新能力**：系统当前仅有客户端 IP 提取（[ip.go](file:///d:/Users/lfh/source/repos/threerouter/threerouter-sub2api/backend/internal/pkg/ip/ip.go)），无 GeoIP。`DATA_RESIDENCY_RISK`/`CROSS_BORDER_TRANSFER`/`SANCTIONED_REGION`/`user_jurisdiction` 均依赖 GeoIP，需引入 `oschwald/maxminddb-golang` + MaxMind GeoLite2（注意 EULA 与库文件更新机制），列为 P0 前置任务。

5. **服务接入遵循 wire 注入惯例**：新 service 定义构造函数 → 加入 `service/wire.go` ProviderSet → `make` 重新生成 wire_gen.go；后台服务在 `cmd/server/wire.go` `provideCleanup` 注册优雅关闭。

6. **前端无 UI 组件库**：使用自研组件（DataTable、BaseDialog、Pagination 等），"拖拽排序"等复杂交互建议简化为数字输入优先级；i18n 新增键需同步补 locale 完整性测试（参照 `riskControlLocales.spec.ts`）。

### 0.4 流式输出审核的现实约束

系统为 **SSE 流式转发**，"实时拦截高风险输出"在流式场景下不可行（输出逐 token 流出，整体审核前无法判定；中途拦截导致客户端收到截断响应）。**输出侧审核策略修正为**：

- **流式**：异步事后审核（流结束后对聚合输出打标、计入违规次数，不做实时拦截），复用现有异步 worker 队列；
- **非流式**：可选同步输出审核；
- 对外**不承诺**"实时拦截流式输出"。

### 0.5 ZDR 与计费事务的字段级策略

现有 UsageLog（含 IP/UA）是**计费的事务性依据**（日志插入与扣费同事务），不能简单"不记"。ZDR 模式采用**字段级策略**：

| 字段类别 | ZDR 模式处理 | 合法性基础 |
|---------|-------------|-----------|
| 计费必需（tokens、成本、model、user_id） | **始终保留** | GDPR Art 6(1)(b) 合同履行 |
| 个人数据（ip_address、user_agent） | 不写入 / 匿名化（IP 截断为 /24 或哈希） | 数据最小化 |
| 内容元数据（prompt_hash、response_hash） | 保留（不可逆） | ZDR 兼容 |

## 二、缺口分析

| 需求 | 现状 | 缺口 |
|-----|------|------|
| API 调用日志与审计追踪 | 仅记录技术层面的使用数据 | 缺少合规维度字段（风险标签），需区分 ZDR/详细日志模式 |
| 内容过滤与拦截 | 已有 ContentModerationService（关键词+OpenAI moderation+入口拦截） | 增量：REGEX/PATTERN 规则、规则 CRUD、多第三方适配、策略组合、输出侧事后审核 |
| 模型风险概览 | 无系统化标签体系 | 需要分离模型属性标签和合规风险标签；GeoIP 相关标签需先引入 GeoIP 能力 |
| EU AI Act 合规评估报告 | 无相关功能 | 需要生成基于 AI 服务链角色、透明度、风险管理的合规评估报告（非 GPAI 技术文档） |
| 跨法域合规映射 | 无相关功能 | 需要支持 EU/中国/加州等多法域合规规则匹配 |
| 行业合规模板 | 无相关功能 | 需要电商、金融、医疗等行业预置合规策略 |
| **GDPR 合规** | 无专门的 GDPR 合规功能 | 需要数据主体权利（删除权、可移植性）、同意管理、数据处理记录 |
| **ZDR（零数据保留）** | UsageLog 记录详细请求数据 | 默认只保留聚合统计，明细日志需用户 opt-in |

### 3.2 ZDR 兼容的数据分层策略

| 层级 | 存储内容 | 默认策略 | 合规说明 |
|-----|---------|---------|---------|
| **Aggregate Layer** | 聚合统计数据（按用户/小时/模型汇总）、不可逆元数据、聚合审计信息 | 默认开启 | ZDR 兼容，不存储个人数据 |
| **Detail Layer** | 单条请求的 IP、模型、tokens、hash、请求元数据等 | 需 opt-in | 非 ZDR 兼容，需用户明确同意 |
| **Compliance Layer** | 合规事件、风险评估、凭证记录 | 始终存储 | 元数据，不包含请求内容 |

**审计能力说明**：提供完整 API 调用审计能力；在 ZDR 模式下保存不可逆元数据和聚合审计信息，在用户授权模式下保存详细调用日志。

### 3.3 ZDR 安全模式 vs Audit 模式

| 模式 | 存储内容 | 适用场景 |
|-----|---------|---------|
| **ZDR 安全模式** | hash、timestamp、token统计、model id | 严格 ZDR 客户，需要证明数据未保留 |
| **Audit 模式** | request metadata、response metadata | 客户授权后，用于问题排查和合规审计 |

**Hash 计算方法**：

```go
import (
    "crypto/sha256"
    "encoding/hex"
)

func ComputeHash(content string) string {
    h := sha256.New()
    h.Write([]byte(content))
    return hex.EncodeToString(h.Sum(nil))
}
```

- `prompt_hash`/`response_hash` 使用 **SHA-256** 计算，取完整 64 位十六进制字符串
- 计算前对内容进行 UTF-8 规范化（NFC），确保相同内容产生一致哈希
- ZDR 模式下仅计算和存储哈希，不存储原始内容

**GeoIP 降级策略**：

```go
type GeoIPService struct {

    fallbackEnabled bool
}

func (g *GeoIPService) Lookup(ip string) (string, error) {
    if g.reader == nil {
        if g.fallbackEnabled {
            return "", nil 
        }
        return "", fmt.Errorf("geoip not available")
    }
    
    var record struct {
        Country struct {
            ISOCode string `maxminddb:"iso_code"`
        } `maxminddb:"country"`
    }
    
    ipAddr := net.ParseIP(ip)
    if err := g.reader.Lookup(ipAddr, &record); err != nil {

            return "", nil
        }
        return "", err
    }
    
    return record.Country.ISOCode, nil
}
```

**策略控制实现**：

```go
// 策略管理器
type StrategyManager struct {
    rules      map[string]*ModerationRule
    strategies map[string]*ModerationStrategy
    detector   *RiskDetector
}

func (sm *StrategyManager) Evaluate(ctx context.Context, input string, strategyID string) (*ModerationResult, error) {
    // 1. 获取策略
    strategy := sm.strategies[strategyID]
    
    // 2. 执行规则匹配
    matchedRules := []string{}
    totalScore := 0.0
    
    for _, ruleID := range strategy.Rules {
        rule := sm.rules[ruleID]
        if !rule.enabled {
            continue
        }
        
        // 执行规则检测
        score, matched := sm.detector.Detect(ctx, input, rule)
        if matched {
            matchedRules = append(matchedRules, ruleID)
            totalScore += score
        }
    }
    
    // 3. 综合评分
    finalScore := totalScore / float64(len(strategy.Rules))
    
    // 4. 策略决策
    result := &ModerationResult{
        Confidence:   finalScore,
        MatchedRules: matchedRules,
    }
    
    switch strategy.LogicType {
    case "AND":
        // 所有规则必须匹配
        if len(matchedRules) == len(strategy.Rules) {
            result.Action = strategy.DefaultAction
        } else {
            result.Action = "ALLOW"
        }
    case "OR":
        // 任一规则匹配即触发
        if len(matchedRules) > 0 {
            result.Action = strategy.DefaultAction
        } else {
            result.Action = "ALLOW"
        }
    case "WEIGHTED":
        // 加权评分
        if finalScore >= strategy.Threshold {
            result.Action = strategy.DefaultAction
        } else {
            result.Action = "ALLOW"
        }
    }
    
    return result, nil
}
```

## 九、合规说明

### 9.1 EU AI Act 适用范围

Sub2API 当前定位为 AI 模型访问与服务聚合基础设施，不提供自主 AI 决策能力、不控制最终使用目的、不直接向用户提供特定领域 AI 服务，因此当前不作为 EU AI Act 下的高风险 AI 系统提供者。

系统主要提供 AI 服务链透明度支持能力，包括调用审计、模型信息记录、风险标识和合规证据管理。这些能力并非用于证明 Sub2API 自身的合规状态，而是帮助使用 Sub2API 的企业客户完成他们自己的 AI 治理和审计需求。

**商业定位**：Sub2API 不一定是被监管的 AI 系统，但它可以成为被监管 AI 系统的合规基础设施。

若未来扩展为面向终端用户的 AI 应用、自动决策系统或特定高风险领域解决方案，则需重新进行 EU AI Act 角色评估。

### 9.2 GDPR 合规要点

| GDPR 条款 | 实施方案 |
|-----------|---------|
| **Art 5(1)(c) 数据最小化** | Aggregate-only 模式默认开启，详细日志仅用户 opt-in |
| **Art 7 同意管理** | UserConsent 表记录所有同意状态，支持随时撤销 |
| **Art 17 删除权** | DataErasureRequest 处理流程，支持 FULL_ERASURE/ANONYMIZE/RESTRICT |
| **Art 20 数据可移植性** | 用户数据导出接口，支持 JSON 格式导出 |
| **Art 30 数据处理记录** | 支持生成数据处理活动记录（RoPA）辅助材料，可包含客户配置的隐私联系人/DPO信息。 |
| **Art 32 安全措施** | 审计日志加密存储，访问控制 |
| **Art 33 数据泄露通知** | 新增 breach log + 通知模板 |
| **Art 46 跨境传输** | 当前系统部署于美国区域。如客户处理 EU/EEA 用户个人数据，需要确认是否存在 GDPR Chapter V 跨境传输要求，并根据客户情况采用 SCC、EU-US DPF（如适用）或其他合法传输机制。 |

### 9.3 ZDR 合规要点

- 默认只存储聚合统计数据，不存单条请求明细
- 详细日志需用户明确 opt-in，可配置保留期（7/30/90天）
- 保留到期后自动清理明细数据
- 合规证据包和审计事件不包含请求内容
- 提供 ZDR 安全模式和 Audit 模式两种选择

### 9.4 数字服务出口合规证据包

按照中国商务部数字服务贸易相关统计要求，用于生成企业申报和审计所需的辅助材料。要求设计，包含：
- 服务类型和范围
- 服务周期
- 交易记录摘要
- 用户区域统计
- 数据流向说明
- 合规声明
- 审计哈希

**重要说明**：本系统提供的是合规证明材料，而非官方凭证。材料包可用于企业内部合规管理和商务申报参考。

### 9.5 数据不可篡改

所有审计日志和证据包记录采用追加式写入，确保合规数据的完整性和可追溯性。
