## 二、缺口分析

| 需求 | 现状 | 缺口 |
|-----|------|------|
| 全量审计日志 | 仅记录技术层面的使用数据 | 缺少合规维度字段（风险标签、数据驻留、AI Act分类） |
| 风险标签 | 无系统化标签体系 | 需要定义风险标签分类和自动化打标机制 |
| EU AI Act 报告 | 无相关功能 | 需要根据 AI Act Title IV 生成透明度报告 |
| 数字服务出口凭证 | 无相关功能 | 需要生成符合商务部要求的可申报凭证 |
| **GDPR 合规** | 无专门的 GDPR 合规功能 | 需要数据主体权利（删除权、可移植性）、同意管理、数据处理记录 |
| **ZDR（零数据保留）** | UsageLog 记录详细请求数据 | 默认只保留聚合统计，明细日志需用户 opt-in |

## 三、合规框架设计

### 3.1 合规目标

```
┌─────────────────────────────────────────────────────────┐
│                     合规模块架构                         │
├─────────────────────────────────────────────────────────┤
│  外部合规要求                                            │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │  GDPR    │ │ EU AI    │ │ 数字服务  │ │  ZDR     │   │
│  │          │ │   Act    │ │ 出口凭证  │ │          │   │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘   │
│       │            │            │            │          │
├───────┼────────────┼────────────┼────────────┼──────────┤
│  合规核心层                                             │
│  ┌───────────────────────────────────────────────────┐ │
│  │  数据分层存储  │  风险评估  │  凭证管理  │  报告生成  │ │
│  │  Aggregate    │  Engine    │  Service   │  Service  │ │
│  │  + Detail     │            │            │           │ │
│  └───────────────────────────────────────────────────┘ │
├───────┴────────────┴────────────┴────────────┴──────────┤
│  数据存储层                                             │
│  ┌──────────┐ ┌─────────────┐ ┌─────────────┐ ┌──────┐ │
│  │ UsageLog │ │Compliance   │ │Compliance   │ │Data  │ │
│  │ (分层)   │ │AuditLog     │ │Credential   │ │Erasure││
│  └──────────┘ └─────────────┘ └─────────────┘ └──────┘ │
└─────────────────────────────────────────────────────────┘
```

### 3.2 ZDR 兼容的数据分层策略

| 层级 | 存储内容 | 默认策略 | 合规说明 |
|-----|---------|---------|---------|
| **Aggregate Layer** | 聚合统计数据（按用户/小时/模型汇总） | 默认开启 | ZDR 兼容，不存储个人数据 |
| **Detail Layer** | 单条请求的 IP、模型、tokens 等 | 需 opt-in | 非 ZDR 兼容，需用户明确同意 |
| **Compliance Layer** | 合规事件、风险评估、凭证记录 | 始终存储 | 元数据，不包含请求内容 |

#### 4.1.2 风险标签分类体系

| 标签 | 说明 | 触发条件 |
|-----|------|---------|
| `PII_DETECTED` | 检测到个人身份信息 | 输入文本包含手机号、邮箱、身份证等 |
| `HIGH_RISK_MODEL` | 高风险模型调用 | 调用 GPT-4o、Claude Opus 等高级模型 |
| `CROSS_BORDER_DATA` | 跨境数据传输 | 用户区域与部署节点不在同一司法管辖区 |
| `SANCTIONED_REGION` | 制裁区域访问 | 用户IP来自受制裁国家/地区 |
| `CONTENT_POLICY_VIOLATION` | 内容政策违规 | 内容审核未通过 |
| `RATE_LIMIT_EXCEEDED` | 超限调用 | API调用频率超过限制 |
| `ANOMALOUS_BEHAVIOR` | 异常行为 | 行为模式异常（如突然大量调用） |

#### 4.3.2 凭证类型说明

| 凭证类型 | 用途 | 合规依据 |
|---------|------|---------|
| `DIGITAL_SERVICE_EXPORT` | 数字服务出口申报凭证 | 中国商务部《数字服务贸易统计制度》 |
| `EU_AI_ACT_COMPLIANCE` | EU AI Act 合规声明凭证 | EU AI Act Title IV |
| `GDPR_COMPLIANCE` | GDPR 合规声明凭证 | GDPR Art 30 |
| `ZDR_CERTIFICATION` | 零数据保留认证凭证 | ZDR 承诺 |

#### 4.3.3 凭证生成流程

```
用户请求 → 合规检查 → 风险评估 → 聚合数据验证 → 生成凭证 → 签名存储 → 返回凭证ID
```

## 九、合规说明

### 9.1 EU AI Act 适用范围

本系统作为 API 中继服务，属于 **Limited Risk（Title IV）**，需履行透明度义务，无需进行高风险 AI 系统的全面合规评估。

### 9.2 GDPR 合规要点

| GDPR 条款 | 实施方案 |
|-----------|---------|
| **Art 5(1)(c) 数据最小化** | Aggregate-only 模式默认开启，明确日志仅用户 opt-in |
| **Art 7 同意管理** | UserConsent 表记录所有同意状态，支持随时撤销 |
| **Art 17 删除权** | DataErasureRequest 处理流程，支持 FULL_ERASURE/ANONYMIZE/RESTRICT |
| **Art 20 数据可移植性** | 用户数据导出接口，支持 JSON 格式导出 |
| **Art 30 数据处理记录** | 自动生成数据处理活动记录，包含 DPO 联系方式 |
| **Art 32 安全措施** | 审计日志加密存储，访问控制 |
| **Art 33 数据泄露通知** | 新增 breach log + 通知模板 |
| **Art 46 跨境传输** | DPA 自动生成、SCCs 声明 |

### 9.3 ZDR 合规要点

- 默认只存储聚合统计数据，不存单条请求明细
- 详细日志需用户明确 opt-in，可配置保留期（7/30/90天）
- 保留到期后自动清理明细数据
- 合规凭证和审计事件不包含请求内容

### 9.4 数字服务出口凭证

按照中国商务部《数字服务贸易统计制度》要求设计，包含：
- 服务类型和范围
- 交易金额和结算方式
- 境外消费者所在国家/地区
- 服务提供方式（跨境交付）

### 9.5 数据不可篡改

所有审计日志和凭证记录采用追加式写入，确保合规数据的完整性和可追溯性。

## 十、与客户确认的关键问题

在实施前需与客户确认以下问题：

1. **ZDR 的具体范围**：
   - Level 1（不存请求内容）→ 当前方案可满足
   - Level 2（不存单条请求明细）→ 需要实施数据分层设计
   - Level 3（完全不留日志）→ 与审计日志需求矛盾

2. **数据驻留偏好**：
   - 当前系统走美东/美西节点，对 GDPR 的 EU-US 数据传输是否有特殊要求？（SCCs / DPF）

3. **日志保留期**：
   - 如果接受 opt-in 的详细日志，默认保留期需要多久？（7天/30天/90天？）

4. **DPO 联系方式**：
   - 是否需要在合规页面展示 DPO 联系方式？

5. **数据处理协议（DPA）**：
   - 是否需要支持生成标准的 DPA 文档？
